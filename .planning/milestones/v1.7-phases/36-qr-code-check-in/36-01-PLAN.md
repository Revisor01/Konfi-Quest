**Backend (backend/routes/events.js):**

1. QR_SECRET definieren am Anfang der Route-Datei (nach jwt require):
```javascript
const QR_SECRET = process.env.QR_SECRET || process.env.JWT_SECRET;
```

2. **POST /:id/generate-qr** (requireTeamer):
   - Event laden: `SELECT id, qr_token FROM events WHERE id = $1 AND organization_id = $2`
   - Wenn qr_token bereits existiert: direkt zurueckgeben `{ qr_token: event.qr_token }`
   - Sonst: Token generieren mit `jwt.sign({ eid: parseInt(id), oid: req.user.organization_id }, QR_SECRET, { algorithm: 'HS256' })` - KEIN expiresIn (Zeitfenster laeuft ueber event_date)
   - Token in DB speichern: `UPDATE events SET qr_token = $1 WHERE id = $2`
   - Response: `{ qr_token: token }`

3. **POST /qr-checkin** (rbacVerifier, KEIN requireTeamer - Konfis muessen zugreifen koennen):
   - Token aus Body: `const { token } = req.body`
   - Token verifizieren: `jwt.verify(token, QR_SECRET)` -> catch: 400 "Ungueltiger QR-Code"
   - Event laden mit qr_token-Abgleich: `SELECT id, name, event_date, checkin_window, mandatory, points, point_type, qr_token, organization_id FROM events WHERE id = $1 AND qr_token = $2`
   - Organization-Check: event.organization_id === req.user.organization_id, sonst 403
   - **Zeitfenster-Pruefung** (komplett in PostgreSQL fuer korrekte Zeitzonen):
     ```sql
     SELECT NOW() BETWEEN (event_date - ($1 || ' minutes')::interval) AND (event_date + ($1 || ' minutes')::interval) AS in_window,
            NOW() < (event_date - ($1 || ' minutes')::interval) AS too_early,
            NOW() > (event_date + ($1 || ' minutes')::interval) AS too_late
     FROM events WHERE id = $2
     ```
   - Bei too_early: 400 `{ error: 'Check-in ist noch nicht moeglich', error_type: 'too_early', event_date, checkin_window }`
   - Bei too_late: 400 `{ error: 'Der Check-in-Zeitraum ist abgelaufen', error_type: 'too_late' }`
   - Booking pruefen: `SELECT id, status, attendance_status FROM event_bookings WHERE event_id = $1 AND user_id = $2`
   - Kein Booking: 400 `{ error: 'Du bist nicht fuer dieses Event angemeldet', error_type: 'not_registered' }`
   - status === 'opted_out': 400 `{ error: 'Du hast dich von diesem Event abgemeldet', error_type: 'opted_out' }`
   - status !== 'confirmed': 400 `{ error: 'Deine Anmeldung ist nicht bestaetigt', error_type: 'not_confirmed' }`
   - attendance_status === 'present': 200 `{ message: 'Du bist bereits eingecheckt', already_checked_in: true, event_name: event.name, event_id: event.id }`
   - **Attendance setzen mit Punkte-Vergabe** (Transaction, gleiche Logik wie PUT /:id/participants/:participantId/attendance):
     - BEGIN Transaction mit dediziertem Client
     - UPDATE event_bookings SET attendance_status = 'present' WHERE id = booking.id
     - Wenn points > 0 und !mandatory: checkPointTypeEnabled, INSERT INTO event_points (ON CONFLICT DO NOTHING), Profile-Update, Badge-Check
     - COMMIT
     - Nach COMMIT: Push (sendEventAttendanceToKonfi), LiveUpdate
   - Response: `{ message: 'Erfolgreich eingecheckt', event_name: event.name, event_id: event.id, points_awarded: boolean }`

4. **GET /:id/attendance-count** (requireTeamer):
   - Query: `SELECT COUNT(*) FILTER (WHERE attendance_status = 'present') AS checked_in, COUNT(*) AS total FROM event_bookings WHERE event_id = $1 AND status = 'confirmed'` (organization_id via JOIN pruefen)
   - Response: `{ checked_in: number, total: number }`

5. **GET-Queries erweitern:**
   - In GET / (Event-Liste) und GET /:id (Event-Detail): `checkin_window` Feld zum SELECT hinzufuegen
   - In POST und PUT: `checkin_window` aus req.body akzeptieren und speichern (Integer, Default 30, min 5, max 120)
