Mechanische Migration per D-12, D-13, D-14, D-15. Jede Page folgt das gleiche Pattern:

**Migration-Pattern (fuer alle Pages identisch):**
1. Import hinzufuegen: `import { useOfflineQuery } from '../../../hooks/useOfflineQuery';` und `import { CACHE_TTL } from '../../../services/offlineCache';`
2. `useState` + `loadData` + `useEffect(() => loadData())` ersetzen durch `useOfflineQuery`
3. `useLiveRefresh('type', loadData)` → `useLiveRefresh('type', refresh)`
4. IonRefresher-Handler: `await refresh(); event.detail.complete();`
5. Window Event Listener (`events-updated` etc.) bleiben erhalten, rufen jetzt `refresh()` statt `loadData()` auf

**KonfiDashboardPage** (komplexeste — 4 separate useOfflineQuery, per D-15):
- `api.get('/konfi/dashboard')` → `useOfflineQuery('konfi:dashboard:' + user?.id, () => api.get('/konfi/dashboard').then(r => r.data), { ttl: CACHE_TTL.DASHBOARD })`
- `api.get('/konfi/tageslosung')` → `useOfflineQuery('konfi:tageslosung:' + new Date().toISOString().split('T')[0], fetcher, { ttl: CACHE_TTL.TAGESLOSUNG })`
- `api.get('/konfi/events')` → `useOfflineQuery('konfi:events:' + user?.id, ..., { ttl: CACHE_TTL.EVENTS, select: filterUpcoming })`
- `api.get('/konfi/badges')` → `useOfflineQuery('konfi:badges:' + user?.id, ..., { ttl: CACHE_TTL.BADGES })`
- `loading` wird nur vom Dashboard-Query bestimmt: `const loading = dashLoading;`
- useLiveRefresh(['dashboard', 'events', 'badges'], refreshAll) — refreshAll ruft alle 3 refresh() Funktionen auf
- Tageslosung-Transformation (replace regex) geht in den fetcher

**KonfiEventsPage** (1 Call):
- Cache-Key: `konfi:events:${user?.id}`, TTL: CACHE_TTL.EVENTS
- Window Event `events-updated` Listener bleibt, ruft `refresh()` auf

**KonfiBadgesPage** (2 Calls via Promise.all):
- Aufteilen in 2 separate useOfflineQuery: badges + profile
- Cache-Keys: `konfi:badges:${user?.id}` (BADGES TTL), `konfi:profile:${user?.id}` (PROFILE TTL)
- `POST /konfi/badges/mark-seen` bleibt als normaler api.post() (Write-Aktion, nicht cachen)
- loading = badgesLoading (Badges bestimmt Loading)

**KonfiRequestsPage** (1 Call):
- Cache-Key: `konfi:requests:${user?.id}`, TTL: CACHE_TTL.REQUESTS
- DELETE-Aktion bleibt als api.delete(), danach `refresh()` aufrufen

**KonfiProfilePage** (1 Call):
- Cache-Key: `konfi:profile:${user?.id}`, TTL: CACHE_TTL.PROFILE

**KonfiEventDetailPage** (1 Call — Details fuer einzelnes Event):
- Pruefen ob diese Page einen eigenen API-Call hat oder Daten von der Events-Page nutzt
- Falls eigener Call: Cache-Key `konfi:event-detail:${eventId}`, TTL: CACHE_TTL.EVENTS
- Write-Aktionen (buchen, abmelden, opt-out) bleiben als normale api-Calls

Chat hat ein Hybrid-Pattern: useOfflineQuery fuer den initialen Load, WebSocket fuer Live-Updates.

**ChatOverview** (Raumliste):
- `api.get('/chat/rooms')` → `useOfflineQuery('chat:rooms:' + user?.id, () => api.get('/chat/rooms').then(r => r.data), { ttl: CACHE_TTL.CHAT_ROOMS })`
- WebSocket-Listener (`newMessage`, `connect`) bleiben VOLLSTAENDIG erhalten
- WebSocket-Listener rufen `refresh()` auf statt die alte loadRooms() Funktion
- Badge-Count-Updates (BadgeContext) bleiben unveraendert

**ChatRoom** (Nachrichten eines Raums):
- `api.get('/chat/rooms/:id/messages?limit=100')` → `useOfflineQuery('chat:messages:' + roomId, () => api.get(...).then(r => r.data), { ttl: CACHE_TTL.CHAT_MESSAGES, enabled: !!roomId })`
- WebSocket-Listener fuer `newMessage`, `messageDeleted`, `reactionAdded`, `reactionRemoved` bleiben VOLLSTAENDIG erhalten
- WebSocket-Listener aktualisieren den lokalen State UND rufen NICHT refresh() auf (wuerde Flackern verursachen)
- Stattdessen: WebSocket-Events aktualisieren `data` lokal via setMessages, und der Cache wird beim naechsten Mount/Refresh aktualisiert
- 30s-Polling-Backup (falls vorhanden) bleibt erhalten
- Write-Aktionen (POST message, Reaktionen) bleiben als normale api-Calls
- WICHTIG: ChatRoom hat komplexen State (Nachrichten werden live hinzugefuegt). useOfflineQuery liefert nur den initialen Load. Danach wird der State lokal verwaltet. Pattern: `const { data: initialMessages, ... } = useOfflineQuery(...)` + `const [messages, setMessages] = useState([])` + useEffect der initialMessages in messages kopiert.

ACHTUNG: Chat-WebSocket-Logik ist komplex und sensibel. Nur den initialen Load migrieren, NICHT die Live-Update-Logik anfassen.
