1. **validate-invite Route (GET /validate-invite/:code) ueberarbeiten:**
   - Die Query aufteilen: Zuerst Code in invite_codes suchen (ohne expires_at/used_at Filter)
   - Wenn Code nicht existiert: `{ error: 'Dieser Einladungscode existiert nicht', error_code: 'not_found' }` mit Status 404
   - Wenn Code abgelaufen (expires_at <= NOW()): `{ error: 'Dieser Einladungscode ist abgelaufen. Bitte frage deinen Konfi-Leiter nach einem neuen Code.', error_code: 'expired' }` mit Status 410
   - Wenn Code gueltig: Bisherige Success-Response beibehalten
   - WICHTIG: Die `used_at IS NULL` Bedingung komplett entfernen -- Codes sind multi-use (per User Decision)

2. **register-konfi Route ueberarbeiten:**
   - Gleiche differenzierte Fehlerbehandlung fuer invite_code wie bei validate-invite
   - Die `used_at IS NULL` Bedingung aus der Query entfernen (Codes sind multi-use)
   - Bei konfi_profiles INSERT: `invite_code_id` Spalte hinzufuegen (Wert: invite.id)
     ```sql
     INSERT INTO konfi_profiles (user_id, jahrgang_id, gottesdienst_points, gemeinde_points, invite_code_id)
     VALUES ($1, $2, 0, 0, $3)
     ```
   - Nach erfolgreicher Registrierung: JWT Token generieren und in Response zurueckgeben (gleiche jwt.sign Logik wie beim Login, mit `expiresIn: '90d'`)
   - Response-Format erweitern:
     ```json
     { "message": "Registrierung erfolgreich", "token": "...", "user": { "id": ..., "display_name": ..., "username": ..., "type": "konfi", "jahrgang": ..., "role_name": "konfi" } }
     ```

   - Validation: Username mindestens 3 Zeichen, nur alphanumerisch + Unterstrich (body-Validator nicht noetig da GET mit param)
