<task type="auto">
  <name>Task 1: Badge-Logik fuer deaktivierte Punkte-Typen anpassen</name>
  <files>backend/routes/badges.js</files>
  <action>
In `backend/routes/badges.js`, Funktion `checkAndAwardBadges`:

1. **Jahrgang-Config laden** - Am Anfang der Funktion, nach dem Laden der Konfi-Daten, eine Query hinzufuegen:
   ```sql
   SELECT j.gottesdienst_enabled, j.gemeinde_enabled
   FROM konfi_profiles kp
   JOIN jahrgaenge j ON kp.jahrgang_id = j.id
   WHERE kp.user_id = $1
   ```
   Ergebnis als `jahrgangConfig` speichern.

2. **4 Kriterien-Typen im switch/if anpassen:**

   - `gottesdienst_points`: Wenn `!jahrgangConfig?.gottesdienst_enabled` dann `earned = false` (Badge kann nicht verdient werden, Kriterium wird uebersprungen)
   - `gemeinde_points`: Wenn `!jahrgangConfig?.gemeinde_enabled` dann `earned = false`
   - `both_categories`: Wenn `!jahrgangConfig?.gottesdienst_enabled || !jahrgangConfig?.gemeinde_enabled` dann `earned = false` (braucht BEIDE aktiv)
   - `total_points`: Nur aktive Typen summieren:
     ```javascript
     let total = 0;
     if (jahrgangConfig?.gottesdienst_enabled) total += konfi.gottesdienst_points;
     if (jahrgangConfig?.gemeinde_enabled) total += konfi.gemeinde_points;
     earned = total >= badge.criteria_value;
     ```

3. Die anderen Kriterien-Typen (activities_count, etc.) bleiben unveraendert.

WICHTIG: Wenn jahrgangConfig null ist (kein Jahrgang zugewiesen), sicherheitshalber alle Punkte-Kriterien auf earned = false setzen.

  <files>backend/routes/konfi.js</files>
  <action>
In `backend/routes/konfi.js` alle 3 Ranking-Queries anpassen:

1. **Dashboard Top-3 Ranking (Zeile ~98):** Die Punkte-Berechnung `(kp.gottesdienst_points + kp.gemeinde_points) as points` ersetzen durch:
   ```sql
   (CASE WHEN j.gottesdienst_enabled THEN kp.gottesdienst_points ELSE 0 END
   + CASE WHEN j.gemeinde_enabled THEN kp.gemeinde_points ELSE 0 END) as points
   ```
   JOIN auf jahrgaenge hinzufuegen falls nicht vorhanden: `JOIN jahrgaenge j ON kp.jahrgang_id = j.id`

2. **Dashboard User-Ranking mit RANK() (Zeile ~127):** Gleiche CASE-WHEN-Logik anwenden. Die RANK() OVER (ORDER BY ...) muss die gleiche Berechnung verwenden.

3. **Profil-Seite Ranking (Zeile ~400):** Gleiche Anpassung.

Alle 3 Queries muessen den JOIN auf jahrgaenge haben und die CASE-WHEN-Logik fuer die Punkte-Summe verwenden. Die jahrgaenge-Tabelle wird ueber `kp.jahrgang_id = j.id` verknuepft.

HINWEIS: Die bestehende WHERE-Klausel `kp.jahrgang_id = $1` bleibt - alle Konfis im gleichen Jahrgang haben die gleiche Config, aber der CASE WHEN ist trotzdem noetig fuer korrekte Semantik.
