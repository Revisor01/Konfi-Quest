Neuen Service `frontend/src/services/writeQueue.ts` erstellen. Per D-01, D-02, D-03, D-04, D-05:

1. QueueItem Interface definieren (per D-02):
```typescript
interface QueueItem {
  id: string;           // UUID
  method: 'POST' | 'PUT' | 'DELETE';
  url: string;
  body?: any;
  headers?: Record<string, string>;
  maxRetries: number;   // default 5
  retryCount: number;   // starts at 0
  createdAt: number;    // Date.now()
  hasFileUpload: boolean; // true wenn Datei-Upload noetig
  metadata: {
    type: 'chat' | 'request' | 'opt-out' | 'fire-and-forget' | 'admin' | 'teamer';
    clientId: string;   // UUID fuer Idempotency
    roomId?: number;    // fuer Chat
    label?: string;     // Menschenlesbarer Name fuer Fehler-Toast
  };
}
```

2. Persistenz-Layer mit Capacitor Preferences:
- Key: `queue:items` — JSON-Array aller QueueItems
- `_load()`: Liest und parsed Queue aus Preferences
- `_save(items)`: Schreibt Queue in Preferences
- In-Memory-Cache (`_items: QueueItem[]`) fuer schnellen Zugriff, sync mit Preferences

3. Oeffentliche API (per D-03):
- `enqueue(item: Omit<QueueItem, 'id' | 'retryCount' | 'createdAt'>): Promise<QueueItem>` — Generiert UUID id, setzt retryCount=0, createdAt=Date.now(), speichert persistent. Gibt das erstellte Item zurueck.
- `flush(): Promise<FlushResult>` — Arbeitet Items FIFO-sequentiell ab. Nutzt api[method](url, body, {headers}) fuer jeden Request. Bei Erfolg: Item entfernen. Bei 4xx (ausser 408, 429): Item entfernen, zu failed hinzufuegen. Bei 5xx/408/429: retryCount++, wenn retryCount >= maxRetries dann entfernen und zu failed. Speichert nach jedem Item (nicht erst am Ende). Gibt `{ succeeded: QueueItem[], failed: FailedQueueItem[] }` zurueck.
- `flushTextOnly(): Promise<FlushResult>` — Wie flush(), aber ueberspringt Items mit `hasFileUpload === true` (per D-17).
- `remove(id: string): Promise<void>` — Entfernt ein Item nach ID
- `getAll(): Promise<QueueItem[]>` — Gibt alle Items zurueck
- `getByMetadata(filter: Partial<QueueItem['metadata']>): Promise<QueueItem[]>` — Filtert Items nach metadata-Feldern
- `clear(): Promise<void>` — Leert gesamte Queue

4. FlushResult und FailedQueueItem Interface:
```typescript
interface FailedQueueItem extends QueueItem {
  error: { status: number; message: string; };
}
interface FlushResult {
  succeeded: QueueItem[];
  failed: FailedQueueItem[];
}
```

5. Auto-Flush bei Online-Wechsel:
- Im Modul-Init: `networkMonitor.subscribe((isOnline) => { if (isOnline) flush(); })` aufrufen
- Flush nur starten wenn nicht bereits ein Flush laeuft (`_flushing` Flag)

6. handleFlushResult Helper (intern):
- Fuer jedes failed Item: IonToast oder console.warn mit `item.metadata.label`
- Nutze `import { toastController } from '@ionic/core'` fuer Toast-Anzeige bei failed Items: `"{label} konnte nicht gesendet werden"`

WICHTIG: Kein `_initialized` Pattern noetig — der subscribe-Call passiert beim Module-Load. Der In-Memory-Cache wird beim ersten Zugriff lazy geladen.
