    Schritt 1 (Spalte): `ALTER TABLE events ADD COLUMN IF NOT EXISTS is_konfirmation BOOLEAN DEFAULT false;` — idempotent.

    Schritt 2 (Datenueberfuehrung): Alle Events, die ueber event_categories mit einer Kategorie verknuepft sind, deren name per ILIKE '%konfirmation%' matcht, auf is_konfirmation=true setzen:

    Mehrere Events koennen betroffen sein (kein Unique-Constraint, KONF-04). Bei leerer/ohne-Match-DB trifft das UPDATE 0 Zeilen (Test-DB-sicher).

    Schritt 3 (Verknuepfungen loesen): Erst die event_categories-Verknuepfungen zu den Konfirmation-Kategorien entfernen, damit kein FK die Kategorie-Loeschung blockiert:
    `DELETE FROM event_categories ec USING categories c WHERE ec.category_id = c.id AND c.name ILIKE '%konfirmation%';`

    Schritt 4 (Kategorie loeschen): Die Konfirmation-Kategorie(n) selbst entfernen, damit sie nicht mehr waehlbar/anlegbar im UI erscheinen (KONF-07):
    `DELETE FROM categories WHERE name ILIKE '%konfirmation%';`

    Reihenfolge strikt einhalten (UPDATE vor DELETE event_categories vor DELETE categories), sonst gingen die Match-Informationen verloren. Alle Schritte idempotent/wiederholungssicher: erneuter Lauf setzt nichts faelschlich zurueck (Flag bleibt true; bereits geloeschte Kategorien matchen nicht mehr). KEIN Datenverlust an Events (KONF-08): nur Kategorie-Verknuepfungen und die Kategorie selbst werden entfernt, das Event bleibt mit gesetztem Flag bestehen.
