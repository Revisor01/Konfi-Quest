    - Response enthaelt category_data (STRING_AGG Ergebnis)

    - Teamer bekommt 403

    **3. PUT /api/admin/activities/:id (requireAdmin):**
    - Admin aktualisiert bestehende Activity (id=1) -> 200
    - Nicht-existierende ID -> 404
    - Admin aus Org 2 kann Activity aus Org 1 NICHT aendern -> 404 (Org-Filter)

    **4. DELETE /api/admin/activities/:id (requireAdmin):**
    - Admin loescht Activity -> 200
    - Admin aus Org 2 kann Activity aus Org 1 NICHT loeschen -> 404

    **5. POST /api/admin/activities/assign-activity (requireTeamer, per D-06):**
    - Teamer weist Activity an Konfi zu: { konfiId: 1, activityId: 1, date: '2026-03-28' } -> 200/201
    - Danach: SELECT konfi_profiles WHERE user_id=1 und pruefe dass gottesdienst_points um Activity.gp gestiegen ist
    - Konfi bekommt 403 auf assign-activity

    **6. Kategorie-Filter (per D-07):**
    - GET /api/admin/activities?target_role=konfi prueft Filterung

    Falls globalSetup.js fehlende Spalten hat (z.B. activities.target_role, activity_categories Tabelle), diese ergaenzen analog zum Pattern aus Phase 102 (ALTER TABLE IF NOT EXISTS).

    WICHTIG: Alle Tests pruefen HTTP Status-Codes UND Response-Body-Struktur (per D-04).
