  <action>
    Die Badge-Query in `updateAllUserBadges` (Zeilen 58-66) erweitern:

    1. Ersetze die einfache Chat-only Query durch eine kombinierte Query die 3 Werte liefert:
       - `chat_unread`: Wie bisher (ungelesene Chat-Nachrichten)
       - `pending_requests`: Nur fuer admin — Anzahl offene Activity-Requests (status = 'pending') in der Organisation des Users
       - `pending_events`: Nur fuer admin — Anzahl Events mit unverarbeiteten Bookings (event_date < CURRENT_DATE, confirmed Bookings ohne attendance_status)

    2. SQL-Query wie in RESEARCH.md Zeile 204-236 beschrieben:
       ```sql
       SELECT
         (SELECT COUNT(DISTINCT cm.id)::int FROM chat_messages cm
          JOIN chat_participants cp ON cm.room_id = cp.room_id
          LEFT JOIN chat_read_status crs ON cm.room_id = crs.room_id
            AND crs.user_id = $1 AND crs.user_type = $2
          WHERE cp.user_id = $1 AND cp.user_type = $2
            AND cm.created_at > COALESCE(crs.last_read_at, '1970-01-01')
            AND cm.deleted_at IS NULL
            AND NOT (cm.user_id = $1 AND cm.user_type = $2)
         ) as chat_unread,
         CASE WHEN $2 = 'admin' THEN (
           SELECT COUNT(*)::int FROM activity_requests ar
           JOIN konfi_profiles kp ON ar.konfi_id = kp.user_id
           JOIN users u ON u.id = $1
           WHERE kp.organization_id = u.organization_id AND ar.status = 'pending'
         ) ELSE 0 END as pending_requests,
         CASE WHEN $2 = 'admin' THEN (
           SELECT COUNT(DISTINCT e.id)::int FROM events e
           JOIN event_bookings eb ON e.id = eb.event_id
           JOIN users u ON u.id = $1
           WHERE e.organization_id = u.organization_id
             AND e.event_date < CURRENT_DATE
             AND eb.status = 'confirmed'
             AND eb.attendance_status IS NULL
         ) ELSE 0 END as pending_events
       ```

    3. WICHTIG: Die aktuelle Query nutzt `cp.last_read_at` — aber die RESEARCH.md Query nutzt `chat_read_status` Tabelle mit LEFT JOIN. Pruefen welche Tabelle tatsaechlich im Schema existiert. Falls `chat_read_status` nicht existiert, bei `cp.last_read_at` bleiben (wie die aktuelle Query).

    4. Badge-Count berechnen: `const badgeCount = (result.chat_unread || 0) + (result.pending_requests || 0) + (result.pending_events || 0)`

    5. Badge Update weiterhin nur senden wenn badgeCount > 0

    6. Badge.clear via Silent Push wenn badgeCount === 0: Aktuell wird bei Count 0 kein Update gesendet. Das ist ok — Badge wird nur gesetzt, nicht aktiv geloescht (Frontend-BadgeContext handled das).

    KEINE neuen Dependencies noetig. Keine Aenderung an der Service-Struktur (startBadgeUpdateService etc. bleiben gleich).
