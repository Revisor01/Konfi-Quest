Erstelle `backend/migrations/064_add_missing_fks.sql` mit fehlenden Foreign Keys.

Bekannte Probleme:
1. **push_tokens.user_id** — hat KEINEN FK auf users (nur "user_id INTEGER NOT NULL" ohne REFERENCES). Ergaenze FK.
2. **notifications** — Tabelle wird in Routes referenziert aber Schema ist nicht in Migrations. Pruefen ob FK auf users existiert.
3. **event_points** — wird in Organizations-Delete referenziert. Pruefen ob FKs auf events und organizations existieren.
4. **password_resets.user_id** — Pruefen ob FK existiert.
5. **chat_read_status** — Pruefen ob FKs auf chat_rooms und users existieren.
6. **chat_polls** — Pruefen ob FK auf chat_rooms existiert.
7. **chat_poll_votes** — Pruefen ob FKs auf chat_polls und users existieren.

Verwende idempotente Syntax mit DO-Block:
```sql
DO $$ BEGIN
  IF NOT EXISTS (SELECT 1 FROM information_schema.table_constraints
    WHERE constraint_name = 'fk_push_tokens_user' AND table_name = 'push_tokens') THEN
    ALTER TABLE push_tokens ADD CONSTRAINT fk_push_tokens_user
      FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE;
  END IF;
END $$;
```

WICHTIG:
- Alle FKs mit ON DELETE CASCADE (da Organization-Delete bereits kaskadierend loescht)
- NUR FKs hinzufuegen wo die referenzierte Tabelle tatsaechlich existiert
- NUR fuer Spalten die TATSAECHLICH referenzieren (user_id -> users.id etc.)
- push_tokens.user_type macht FK kompliziert — push_tokens referenziert user_id aber user_type bestimmt die Quelle. Da alle User jetzt in users-Tabelle sind (RBAC), ist FK auf users korrekt.
