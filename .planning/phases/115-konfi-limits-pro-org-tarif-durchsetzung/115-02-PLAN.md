    - max_konfis NULL -> normal anlegen, 201 (kein Check, D-06).
    - stufe under_limit -> 201 wie bisher.
    - stufe grace UND req.body.confirm !== true -> 409 mit JSON { error, error_code: 'limit_grace',
      count, limit, next_tier }. Kein User wird angelegt (Rollback).
    - stufe grace UND req.body.confirm === true -> 201, Konfi wird angelegt.
    - stufe hard_block -> 403 mit JSON { error, error_code: 'limit_exceeded', count, limit, next_tier }.
      Kein Override, auch nicht mit confirm:true.
    - Bestehende Username-Kollision bleibt 409 mit anderem/keinem error_code (nicht 'limit_grace').
  </behavior>
  <action>
    Importiere checkKonfiLimit in konfi-management.js. Im POST-Handler nach dem jahrgang- und
    role-Check, VOR dem users-INSERT: rufe checkKonfiLimit(client, req.user.organization_id) auf.
    Verzweige nach stufe: hard_block -> ROLLBACK + 403; grace ohne confirm -> ROLLBACK + 409;
    grace mit confirm oder under_limit -> weiter wie bisher. Lies confirm aus req.body.
    Hinweistexte deutsch, echte Umlaute, keine Emojis, z.B. "Tarif ausgeschoepft (X von Y Konfis).
    Du kannst bis zu 5 weitere anlegen — danach ist ein Upgrade noetig." next_tier: naechsthoehere
    Stufe aus der Konstante 15/50/75/100 ableiten (kleinste Stufe > limit; null wenn keine groessere).
    Lege diese Stufen-Konstante als kleines lokales Array/Helper an (Discretion: in konfiLimit.js
    oder lokal). Antworte mit count/limit/next_tier im Body, damit das Frontend (Plan 03) den Dialog
    fuellen kann. error_code-Feld setzen, um den Grace-409 vom Username-409 zu unterscheiden.
  </action>
