  <files>backend/migrations/082_self_auto_deletion.sql</files>
  <action>
Erstelle die Migration `backend/migrations/082_self_auto_deletion.sql` (Nummer 082, weil
081 die letzte vorhandene ist und der Runner alphabetisch sortiert). Die Migration enthaelt
in dieser Reihenfolge:

1. Soft-Delete-Spalten auf users (D-11): `ALTER TABLE users ADD COLUMN IF NOT EXISTS deleted_at TIMESTAMP NULL;`
   und `ALTER TABLE users ADD COLUMN IF NOT EXISTS archived_at TIMESTAMP NULL;`.
   `archived_at` = Zeitpunkt der Soft-Loeschung (Tag 60), `deleted_at` = Marker fuer
   Unsichtbarkeit (wird ab Tag 60 gesetzt). Beide werden genutzt, damit der Cron-Job
   Tag-60-Archivierung und Tag-120-Hard-Delete unterscheiden kann.
2. Partieller Index fuer Filter-Performance: `CREATE INDEX IF NOT EXISTS idx_users_deleted_at ON users (deleted_at) WHERE deleted_at IS NULL;`
3. confirmation_date NOT NULL (D-06) MIT Backfill (Claude's Discretion: Backfill-Strategie):
   Setze fuer bestehende Jahrgaenge ohne Datum einen Fallback, damit der NOT NULL-Constraint
   nicht fehlschlaegt. Strategie: `UPDATE jahrgaenge SET confirmation_date = COALESCE(confirmation_date, created_at::date) WHERE confirmation_date IS NULL;`
   (created_at existiert immer, so bleibt die Auto-Loesch-Frist konservativ und nie in der
   Vergangenheit-springend). Danach `ALTER TABLE jahrgaenge ALTER COLUMN confirmation_date SET NOT NULL;`.

Alle Statements idempotent (IF NOT EXISTS / IF EXISTS wo moeglich). Deutsche Kommentare,
echte Umlaute, keine Emojis. Die ganze Datei wird vom Runner in einem pool.query ausgefuehrt;
gruppiere die Statements als ein SQL-Skript (mehrere Statements durch Semikolon getrennt).
  </action>
